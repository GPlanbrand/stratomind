# Railway 备选部署方案

## 背景

Railway 后端持续构建失败，网络问题无法解决。本文提供三个备选部署方案，包含详细的实施步骤。

---

## 方案 1：Vercel Serverless Functions

### 方案概述

将 Express 后端改造为 Vercel Serverless Functions，使用 `/api` 目录结构。

### 优点

- 与前端同一平台（如果前端也在 Vercel）
- 自动 HTTPS
- 免费额度充足
- 自动预览部署
- Git 集成自动部署

### 缺点

- 需要改造 Express 为 Serverless 格式
- 有 10 秒超时限制
- 无持久连接（WebSocket 需要额外配置）
- cold start 有延迟

### 详细步骤

#### 步骤 1：项目结构改造

将 Express 应用从：

```
my-app/
├── server.js        # Express 入口
├── routes/
│   └── api.js
└── package.json
```

改造为 Serverless 格式：

```
my-app/
├── api/
│   ├── _index.js    # 主页 API
│   ├── users.js     # 用户 API
│   └── posts.js     # 帖子 API
├── package.json
└── vercel.json
```

#### 步骤 2：改造 API 文件

将 `api/users.js` 从 Express 风格：

```javascript
// Express 风格 (server.js)
app.get('/api/users', (req, res) => {
  res.json({ users: [] });
});
```

改造为 Serverless 风格：

```javascript
// api/users.js - Serverless 风格
module.exports = (req, res) => {
  const users = [
    { id: 1, name: 'Alice' },
    { id: 2, name: 'Bob' }
  ];
  res.status(200).json(users);
};
```

或使用新版 Vercel Functions API（推荐）：

```javascript
// api/users.js - 新版 API
export async function GET(request) {
  const users = [
    { id: 1, name: 'Alice' },
    { id: 2, name: 'Bob' }
  ];
  return Response.json(users);
}
```

#### 步骤 3：处理 POST 请求和请求体

```javascript
// api/create.js
module.exports = async (req, res) => {
  if (req.method === 'POST') {
    const { name, email } = req.body;
    // 处理创建逻辑
    res.status(201).json({ message: 'Created', data: { name, email } });
  } else {
    res.status(405).json({ error: 'Method not allowed' });
  }
};
```

#### 步骤 4：创建 vercel.json 配置

```json
{
  "version": 2,
  "builds": [
    { "src": "api/**/*.js", "use": "@vercel/node" }
  ],
  "routes": [
    { "src": "/api/(.*)", "dest": "/api/$1.js" }
  ]
}
```

#### 步骤 5：环境变量配置

在 Vercel Dashboard 中配置：

1. 进入项目 → Settings → Environment Variables
2. 添加环境变量：
   - `DATABASE_URL`
   - `API_SECRET_KEY`
   - `JWT_SECRET`

在代码中使用：

```javascript
const dbUrl = process.env.DATABASE_URL;
```

#### 步骤 6：部署

**方式 A：GitHub 集成（推荐）**

```bash
git init
git add .
git commit -m "Convert to Serverless"
git push origin main
```

然后在 Vercel Dashboard：
1. 访问 vercel.com/new
2. 点击 "Import Git Repository"
3. 选择你的仓库
4. 点击 Deploy

**方式 B：CLI 部署**

```bash
# 安装 Vercel CLI
npm install -g vercel

# 登录
vercel login

# 部署
vercel

# 生产部署
vercel --prod
```

#### 步骤 7：本地测试

```bash
vercel dev
# 访问 http://localhost:3000/api/users
```

## 方案 2：Render.com

### 方案概述

使用 Render 托管 Node.js/Express 应用，提供免费 Web Service。

### 优点

- 原生支持 Express/Node.js，无需改造
- 完全免费（Free 实例）
- 与 Railway 使用体验相似
- 自动从 GitHub 部署
- 提供免费 PostgreSQL

### 缺点

- 15 分钟无活动会进入休眠（冷启动慢）
- 需要新建项目
- 无国内节点

### 详细步骤

#### 步骤 1：准备代码

确保项目根目录有：

```json
// package.json
{
  "name": "my-backend",
  "version": "1.0.0",
  "scripts": {
    "start": "node server.js",
    "dev": "nodemon server.js"
  },
  "dependencies": {
    "express": "^4.18.0",
    "cors": "^2.8.5"
  }
}
```

#### 步骤 2：添加健康检查端点（推荐）

在 `server.js` 中添加：

```javascript
const PORT = process.env.PORT || 3000;

app.get('/health', (req, res) => {
  res.json({ status: 'ok', timestamp: new Date().toISOString() });
});
```

#### 步骤 3：注册并连接 GitHub

1. 访问 https://render.com
2. 点击 "Get Started"
3. 使用 GitHub 账号登录
4. 授权 Render 访问你的仓库

#### 步骤 4：创建 Web Service

1. 在 Render Dashboard 点击 **New +** → **Web Service**
2. 选择你的 GitHub 仓库
3. 配置服务设置：

| 字段 | 值 |
|------|-----|
| Name | `my-backend`（你的应用名） |
| Region | 选择 closest to you |
| Branch | `main` |
| Root Directory | （留空或指定后端目录） |
| Language | `Node` |
| Build Command | `npm install` |
| Start Command | `npm start` |
| Instance Type | **Free** |

#### 步骤 5：配置环境变量

在 Environment Variables 部分添加：

```
DATABASE_URL = your_postgres_connection_string
API_SECRET = your_secret_key
NODE_ENV = production
```

#### 步骤 6：部署

1. 点击 **Create Web Service**
2. 观看构建日志
3. 成功后显示：`==> Your service is live 🎉`
4. 获取 URL：`https://my-backend.onrender.com`

#### 步骤 7：自定义域名（可选）

1. 进入 Service → Settings
2. 滚动到 Custom Domains
3. 添加你的域名
4. 配置 DNS：
   - CNAME: `www` → `onrender.com`
   - 或 A record: `@` → Render 提供的 IP

## 方案 3：Fly.io

### 方案概述

使用 Fly.io Docker 容器托管，提供全球多区域部署。

### 优点

- Docker 容器部署，灵活
- 全球多区域部署
- 免费额度：每月 160GB 带宽，3 个共享 CPU
- 提供免费 PostgreSQL 和 Redis
- 接近原生长时间运行

### 缺点

- 需要注册新账号
- 需要安装 Fly CLI
- Docker 构建时间较长
- 配置相对复杂

### 详细步骤

#### 步骤 1：安装 Fly CLI

**macOS / Linux:**

```bash
curl -L https://fly.io/install.sh | sh
```

**Windows (PowerShell):**

```powershell
pwsh -Command "iwr https://fly.io/install.ps1 -useb | iex"
```

**Homebrew:**

```bash
brew install flyctl
```

#### 步骤 2：登录 Fly.io

```bash
fly auth login
```

#### 步骤 3：准备 Dockerfile

在项目根目录创建 `Dockerfile`:

```dockerfile
# Dockerfile
FROM node:20-slim

WORKDIR /app

# 安装依赖
COPY package*.json ./
RUN npm ci --only=production

# 复制代码
COPY . .

# 暴露端口
EXPOSE 3000

# 启动命令
CMD ["npm", "start"]
```

创建 `.dockerignore`:

```
node_modules
npm-debug.log
.git
.env
```

#### 步骤 4：修改 Express 监听地址

更新 `server.js`:

```javascript
const PORT = process.env.PORT || 3000;

app.listen(PORT, '0.0.0.0', () => {
  console.log(`Server running on port ${PORT}`);
});
```

**重要**：必须监听 `0.0.0.0`，不能是 `localhost`！

#### 步骤 5：初始化 Fly 应用

```bash
fly launch
```

Fly CLI 会询问：

```
App Name: my-backend (或按回车自动生成)
Region: 选择最近的区域 (如 iad for Virginia, sin for Singapore)
PostgreSQL: N
Redis: N
Deploy now: Y
```

这会创建 `fly.toml` 配置文件。

#### 步骤 6：配置环境变量

**方式 A：命令行**

```bash
fly secrets set DATABASE_URL="postgresql://..."
fly secrets set API_SECRET="your-secret"
```

**方式 B：fly.toml**

```toml
[env]
  PORT = "3000"

[env.secrets]
  DATABASE_URL = "postgresql://..."
  API_SECRET = "your-secret"
```

#### 步骤 7：部署

```bash
# 首次部署
fly deploy

# 后续部署
fly deploy

# 查看状态
fly status

# 查看日志
fly logs
```

#### 步骤 8：验证部署

```bash
fly apps list
```

访问：`https://my-backend.fly.dev`

#### 步骤 9：扩展配置（可选）

编辑 `fly.toml`:

```toml
app = "my-backend"

[build]
  dockerfile = "Dockerfile"

[env]
  PORT = "3000"

[http_service]
  internal_port = 3000
  force_https = true
  auto_stop_machines = true
  auto_start_machines = true
  min_machines_running = 0

[[vm]]
  cpu_kind = "shared"
  cpus = 1
  memory_mb = 256
```

## 对比总结

| 特性 | Vercel | Render | Fly.io |
|------|--------|--------|--------|
| 免费额度 | 100GB 带宽/月 | 750小时/月 | 160GB 带宽/月 |
| 冷启动 | 慢 | 很慢 (~15分钟休眠) | 快 |
| Node.js 原生支持 | 需要改造 | ✅ 完全支持 | Docker |
| 持久运行 | ❌ | ❌ | ✅ |
| 国内访问速度 | 一般 | 一般 | 一般 |
| 学习曲线 | 低 | 低 | 中 |
| 部署复杂度 | 中 | 低 | 中高 |

### 推荐优先级

1. **快速迁移**：选择 **Render**（改动最小，原生支持 Express）
2. **长期项目**：选择 **Fly.io**（稳定运行，无冷启动）
3. **与前端同平台**：选择 **Vercel**（前后端统一管理）

---

## 注意事项

1. **数据库迁移**：如果使用 PostgreSQL，建议使用 Render 的免费 Postgres 或 Supabase
2. **环境变量**：所有敏感信息必须在平台环境变量中配置
3. **健康检查**：添加 `/health` 端点便于监控
4. **跨域配置**：确保 CORS 配置允许新的后端域名
5. **前端 API 地址**：部署后需更新前端的 API base URL
